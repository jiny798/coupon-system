쿠폰 발급 동시성 문제 3가지

테스트 결과: 성공 129 / 카운터 63 / 실패 871 — 이 세 숫자가 각각 다른 버그의 증거다

기초 공유락(S) vs 배타락(X)

문제 3(데드락)을 이해하려면 이것부터. DB는 같은 행(row)에 대한 접근을 락으로 조율한다.

공유락 S (Shared) "나 이 행을 읽고 있으니 그동안 아무도 고치지 마."
여럿이 동시에 잡을 수 있다 — 도서관에서 같은 책을 여러 명이 열람하는 것.
배타락 X (Exclusive) "나 이 행을 고치는 중이니 아무도 접근하지 마."
혼자만 잡을 수 있다 — 1인 편집실. 다른 S락, X락 전부 대기.
이미 잡혀있는 락 ↓ \ 새로 요청 →S 요청X 요청
S락 있음✓ 같이 가능✗ 대기
X락 있음✗ 대기✗ 대기

핵심: INSERT issued_coupons는 외래키(coupon_id) 검증을 위해 부모인 쿠폰 행에 S락을 잡고, 카운터를 고치는 UPDATE coupons는 그 행에 X락이 필요하다. 이 조합이 문제 3을 만든다.

정리 세 문제의 공통 뿌리

셋 다 원인은 하나 — "읽고 → 판단/계산하고 → 쓰는" 과정이 원자적이지 않다는 것. 문제 1은 판단이 낡았고, 문제 2는 계산이 낡았고, 문제 3은 그걸 락으로 조율하려다 순서가 꼬인 것이다. 해결책들(synchronized, 비관적/낙관적 락, Redis)은 전부 이 "읽기→쓰기 구간을 어떻게 원자적으로 만들 것인가"에 대한 서로 다른 답이다.
