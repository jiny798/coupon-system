# 4장. 동시성 해결책 비교 — DB 락 3종

> 공통 조건: 재고 100장 / 동시 요청 1,000건(서로 다른 유저) / 스레드 풀 32 / 로컬 MySQL
> 테스트: `CouponIssueConcurrencyTest` (단일 측정값, 정밀 비교 시 3회 중간값 권장)

## 결과 요약

| 방식 | 정합성 | 소요 시간 | 비고 |
|---|---|---|---|
| 무방비 (기준) | ❌ 129장 발급, 카운터 63 | - | 초과 발급 + lost update + 데드락 871건 |
| synchronized (메서드) | ❌ 정확히 2배(200장) 발급 | - | 락 해제(리턴)와 커밋 사이 틈으로 매 값이 2번씩 통과 |
| synchronized (파사드) | ✅ 100 / 100 | 미기록 | 자물쇠가 커밋까지 포함. 단일 JVM 한정, 전역 직렬화 |
| 비관적 락 (FOR UPDATE) | ✅ 100 / 100 | **8,544ms** | 탈락자 900명까지 전원이 행 락에 직렬화 |
| 낙관적 락 (@Version + 재시도) | ✅ 100 / 100 | **7,257ms** (재시도 927회) | 쓰기 경쟁 구간만 비용, 소진 후 거절은 병렬 통과 |

## 핵심 분석

### 왜 낙관적 락이 더 빨랐나 (예상과 반대)

- 이 워크로드는 "쓰기 경쟁 ~10% + 읽고 거절 ~90%" 구조.
- 비관적 락은 이미 소진된 뒤 도착한 900명도 "소진 확인"을 위해 X락 줄에 서야 한다.
- 낙관적 락은 소진 후 트랜잭션이 아무것도 수정하지 않아 version 충돌이 없다
  → 900건의 거절이 락 없이 병렬 처리.
- 재시도 927회는 초반 "100장 채우기" 경쟁 구간에서만 발생 (성공 1건당 약 9회 낭비).

### 결론: 락 선택은 워크로드 모양에 따라 다르다

- 충돌이 드물다 → 낙관적 락 (실패 시에만 비용)
- 충돌이 상시적이고 대부분 쓰기 경쟁 → 비관적 락 (기다렸다 한 번에 성공이 더 쌈)
- 이 테스트는 수량(100) << 요청(1,000)이라 낙관적에 유리. 수량이 요청과 비슷하면 역전 가능.

## 남은 문제 — 다음 장으로

두 방식 모두 정합성은 지켰지만 공통 한계가 있다:

1. **모든 트래픽이 결국 DB에 도달한다.** 발급이든 거절이든 커넥션과 쿼리를 소모한다.
   요청이 1만 RPS면 DB가 1만 RPS를 받아내야 한다.
2. 비관적 락은 대기 중에도 커넥션을 점유 → 인기 쿠폰 하나가 커넥션 풀을 고갈시켜
   무관한 API까지 장애 전파 가능.
3. 낙관적 락은 경쟁이 심해지면 재시도 폭풍으로 DB 부하가 오히려 증폭된다.

→ 다음 장: 경쟁 자체를 DB 앞에서 끝내는 방법 (Redis 원자 연산),
   그리고 DB 쓰기를 뒤로 미루는 방법 (Kafka 비동기) — 원본 블로그의 최종 아키텍처.

## (추가) 해결책 4: Redis 원자 연산 — SADD + INCR

측정 결과: **2,408ms** / 성공 100, 실패 900(전부 CouponSoldOutException) /
Redis 카운트 100 = Redis 유저 셋 100 = DB 발급 내역 100 (3중 일치)

### 갱신된 순위표

| 방식 | 소요 시간 | 배속 (비관적 기준) |
|---|---|---|
| Redis 원자 연산 | **2,408ms** | 3.5x |
| 낙관적 락 (@Version) | 7,257ms | 1.2x |
| 비관적 락 (FOR UPDATE) | 8,544ms | 1.0x |

### 왜 빠른가

- 경쟁 지점(중복 판정, 재고 판정)이 Redis 단일 스레드의 원자 명령 2개로 끝난다.
  락 대기 없음, version 충돌 없음, 재시도 없음.
- 탈락자 900명은 Redis 왕복만으로 밀리초 만에 거절 — DB 쓰기 경로에 진입조차 안 함.
- 공정성 개선: INCR 반환값이 도착 순서 그대로의 순번이라 재시도 복권이 사라짐.

### 남은 비용 (2.4초의 정체)

- 요청 1,000건 전부가 여전히 쿠폰 정의(기간/총수량)를 DB SELECT로 읽는다.
- 당첨자 100명의 INSERT가 여전히 요청 경로 안에서 동기로 실행된다.
- → 개선 여지: 쿠폰 정의 Redis 캐싱, DB 쓰기의 비동기화(다음 장 Kafka).

### 남은 구멍: Redis-DB 정합성

Redis 당첨(INCR ≤ 100)과 DB INSERT 사이에 서버가 죽으면?
→ Redis는 "발급됨"인데 DB엔 기록이 없는 유저 발생 (내 쿠폰 목록에 안 보임).
반대로 DB INSERT 실패 시 Redis를 되돌리는 보상 로직도 현재 없음.
두 저장소에 걸친 작업은 트랜잭션으로 묶을 수 없다 →
"발급 확정 사실"을 유실되지 않는 곳(Kafka)에 기록하고,
DB 반영은 컨슈머가 성공할 때까지 재시도하는 구조로 전환 — 다음 장.
