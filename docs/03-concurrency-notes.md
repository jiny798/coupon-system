# 3장. 동시성 문제 재현 — 학습 노트

> 관련 테스트: `CouponIssueConcurrencyTest` (`./gradlew concurrencyTest`)
> 다이어그램: [concurrency-problems.html](./concurrency-problems.html)

## 실험 결과 (2026-07-21)

| 항목 | 값 |
|---|---|
| 총 수량 | 100 |
| 동시 요청 수 | 1,000 (서로 다른 유저) |
| 성공 응답 | 129 |
| 실패 응답 | 871 |
| issuedQuantity 카운터 | 63 |
| 실제 발급 내역 수 | 129 (**29장 초과 발급**) |

같은 테스트를 다시 돌리면 숫자가 매번 달라진다 → 레이스 컨디션의 비결정성.

## 문제 1. 초과 발급 (129 > 100)

원인: check-then-act. "카운터 읽기 → 100 미만 확인 → 저장" 사이에 다른 트랜잭션이 끼어든다.
A와 B가 둘 다 99를 읽고, 둘 다 통과하고, 둘 다 저장한다.
중복 발급과 달리 재고 체크에는 유니크 제약 같은 DB 레벨 최후 방어선이 없다.

## 문제 2. Lost Update (카운터 63 < 실제 129)

원인: JPA 더티 체킹은 증분(`+1`)이 아니라 **절대값**으로 UPDATE 한다.
`UPDATE coupons SET issued_quantity = 64` — A와 B가 둘 다 63을 읽으면 둘 다 64를 써서
증가 하나가 증발한다. 129번 발급되는 동안 66번 증발했다.
오래된 값을 읽은 트랜잭션이 늦게 커밋하면 카운터가 거꾸로 감소할 수도 있다.

## 문제 3. 데드락 (실패 871건의 정체)

한 트랜잭션의 실제 SQL 순서:

1. `INSERT issued_coupons` — FK(coupon_id) 검증을 위해 부모 쿠폰 행에 **S락** 획득
2. 커밋 직전 `UPDATE coupons` (더티 체킹 flush) — 같은 행에 **X락** 필요

A와 B가 둘 다 1번까지 진행하면 쿠폰 행에 S락 두 개가 공존한다 (S끼리는 호환).
이후 A의 X락 요청은 B의 S락에 막히고, B의 X락 요청은 A의 S락에 막힌다.
서로가 서로의 커밋을 기다리는 사이클 → MySQL 데드락 감지기가 한쪽을 강제 롤백.

## 락 기초 정리

### 공유락 S vs 배타락 X

- **S (Shared)**: "읽는 중이니 고치지 마." 여럿이 동시에 잡을 수 있다 (도서관 열람).
- **X (Exclusive)**: "고치는 중이니 접근하지 마." 혼자만 잡을 수 있다 (1인 편집실).

호환성: S+S ✓ / S+X ✗ / X+X ✗

### 중요한 디테일

- 락은 쿼리가 끝날 때가 아니라 **트랜잭션이 커밋/롤백될 때** 해제된다.
- X락 요청은 거절이 아니라 **대기 큐에 줄을 선다** (`innodb_lock_wait_timeout` 기본 50초).
- **자기가 잡은 S락은 자기 X락 요청을 막지 않는다** (승격 가능).
  남의 S락만 막는다 → "S락 여럿 공존 + 각자 X로 승격 시도" 조합이 데드락을 만든다.
- 데드락은 타임아웃을 기다리지 않는다. 감지 즉시 한쪽을 강제 롤백.

### MVCC — 일반 SELECT는 락을 안 잡는다

InnoDB의 일반 SELECT는 S락 없이 트랜잭션 시작 시점의 **스냅샷(과거 버전)**을 읽는다.
읽기-쓰기가 서로 안 막아서 빠르지만, 그 대가로 낡은 값을 읽는다 (문제 1, 2의 원인).
S락이 걸리는 경우: `SELECT ... FOR SHARE`, 자식 INSERT 시 FK 검증 등.
다음 장에서 배울 `SELECT ... FOR UPDATE`는 읽는 시점부터 X락을 잡는 방식(비관적 락).

## 결론: 세 문제의 공통 뿌리

**"읽고 → 판단/계산하고 → 쓰는" 구간이 원자적이지 않다.**

- 문제 1: 판단의 근거가 낡았다
- 문제 2: 계산의 근거가 낡았다
- 문제 3: 이를 락으로 조율하는 과정에서 획득 순서가 꼬였다

이후 적용할 해결책(synchronized → DB 비관적 락 → 낙관적 락 → Redis 분산락 → Redis INCR)은
전부 "이 구간을 어떻게 원자적으로 만들 것인가"에 대한 서로 다른 답이다.
